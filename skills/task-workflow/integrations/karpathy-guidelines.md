# karpathy-guidelines 集成规范

**版本**：v1.0
**最后更新**：2026-04-15

---

## 一、职责定位

karpathy-guidelines 是 task-workflow 的质量门实现，提供编码规范验证逻辑（类似 Pensieve Pipeline 的 Task 4 验证步骤）。

### 设计参考

**学习对象**：Pensieve Pipeline（`run-when-reviewing-code.md`）

| 设计元素 | Pensieve Pipeline | Karpathy-Guidelines |
|---------|------------------|---------------------|
| 验证逻辑 | Task 4：验证候选问题并过滤误报 | 4 个原则检查 |
| 信号判断 | 置信度 >= 80、高信号问题定义 | 置信度 >= 80、违反原则 <= 2 项 |
| 失败回退 | 证据不足时过滤、无高信号时明确说明 | 违反 <= 2 项重试、>= 3 项停止 |

### 调用机制

- karpathy-guidelines 不是独立技能库，是 task-workflow 的质量门实现
- task-workflow 在代码编写阶段自动调用它
- 调用时机由各 workflow 定义文件指定（见下文"调用时机"）

### 与 pensieve 的协作

```
代码编写完成
    ↓
karpathy-guidelines（细粒度检查）
    ├─ 原则 2: 简洁性
    ├─ 原则 3: 外科手术式
    └─ 原则 4: 验证标准
    ↓ 通过
Pensieve taste review（粗粒度审查，条件触发）
    ├─ 嵌套深度
    ├─ 模块边界
    └─ 接口品味
    ↓ 通过
verification-before-completion
    ↓ 通过
Pensieve self-improve（知识沉淀）
    ├─ 调试策略 → knowledge/
    ├─ 修复选择 → decisions/
    └─ 跨项目铁律 → maxims/
```

### 职责边界

| 技能 | 检查层级 | 调用方式 | 检查对象 |
|------|----------|----------|----------|
| karpathy-guidelines | 代码实现规范 | task-workflow 在代码变更阶段调用 | 函数/模块/文件 |
| pensieve | 架构决策品味 | task-workflow 在关键决策点条件触发 | 系统/架构/接口 |

## 三、数据传递规范

### 输入字段（从上一个 Step 读取）

**代码变更范围**：
- 文件路径1
- 文件路径2

**变更类型**：新增/修改/重构/修复

**实现方案**：代码实现的描述

**成功标准**：可选，如果上一步定义了

### 输出字段（传递给下一个 Step）

**规范检查结果**：通过/不通过

**违反原则**：
- 原则2
- 原则3

**具体问题**：
- 文件：src/auth.py | 行号：45-60 | 问题：添加了未请求的配置系统 | 建议：删除 ConfigManager 类
- 文件：src/utils.py | 行号：12 | 问题：格式化了无关代码 | 建议：回退格式化变更

**置信度**：92%

**修改建议**：移除过度抽象，直接使用环境变量

## 四、质量门标准

### 通过条件

- ✅ 规范检查结果 = "通过"
- ✅ 置信度 >= 80%

### 可修正条件（触发重试）

- ✅ 规范检查结果 = "不通过"
- ✅ 违反原则 <= 2 项
- ✅ 置信度 >= 80%
- ✅ 具体问题有明确修改建议

### 严重失败条件（停止 Workflow）

- ❌ 违反原则 >= 3 项
- ❌ 置信度 < 80%
- ❌ 重试 >= 2 次仍不通过
- ❌ 发现架构级问题（需重新设计）

### 执行逻辑

```
if 规范检查结果 == "通过":
    推进到下一个 Step
elif 规范检查结果 == "不通过" and 违反原则 <= 2 and 置信度 >= 80:
    应用修改建议 → 重新检查（最多 2 次）
else:
    停止 Workflow → 报告给用户
```

## 五、4 个原则的检查逻辑

### 原则 1: Think Before Coding

**检查时机**：计划阶段（writing-plans 之后）

**检查项**：
- ✅ 假设是否显性声明
- ✅ 多种方案是否列出
- ✅ 不确定性是否标注
- ✅ 成功标准是否可验证

**输出格式**：

```markdown
[原则1检查]
- 假设显性化: ✅ 已声明 3 个关键假设
- 方案对比: ⚠️ 仅提供单一方案，建议补充备选方案
- 成功标准: ✅ 可验证（通过单元测试）
```

---

### 原则 2: Simplicity First

**检查时机**：执行阶段（代码编写后）

**检查项**：
- ✅ 是否有未请求的功能
- ✅ 是否有单次使用的抽象
- ✅ 是否有未请求的配置项
- ✅ 是否有不可能场景的错误处理
- ✅ 代码行数是否可压缩

**输出格式**：

```markdown
[原则2检查]
- 未请求功能: ❌ 发现配置系统（未请求）
- 过度抽象: ❌ BaseHandler 仅被继承 1 次
- 代码行数: ⚠️ 200 行可压缩至 ~80 行
- 建议: 删除 ConfigManager，直接使用 os.getenv()
```

---

### 原则 3: Surgical Changes

**检查时机**：执行阶段（代码修改后）

**检查项**：
- ✅ 是否修改了无关代码
- ✅ 是否重构了未损坏的代码
- ✅ 是否匹配现有风格
- ✅ 是否删除了预存在的死代码
- ✅ 每行变更是否可追溯到需求

**输出格式**：

```markdown
[原则3检查]
- 无关修改: ❌ 格式化了 utils.py（与需求无关）
- 风格一致性: ✅ 匹配现有代码风格
- 变更可追溯性: ⚠️ 3 处修改无法追溯到需求
- 建议: 回退 utils.py 的格式化变更
```

---

### 原则 4: Goal-Driven Execution

**检查时机**：计划阶段 + 验证阶段

**检查项**：
- ✅ 成功标准是否可验证
- ✅ 是否有验证命令
- ✅ 多步任务是否有计划
- ✅ 每步是否有检查点

**输出格式**：

```markdown
[原则4检查]
- 成功标准: ✅ "pytest test_auth.py -v"
- 验证命令: ✅ 已提供
- 执行计划: ✅ 3 步计划，每步有检查点
```

## 六、失败回退机制

### 场景 1：规范检查不通过 + 置信度 >= 80%

**动作**：
1. 应用修改建议
2. 重新执行当前 Step
3. 最多重试 2 次

### 场景 2：规范检查不通过 + 置信度 < 80%

**动作**：
1. 报告给用户
2. 等待用户决策（继续/修改/放弃）

### 场景 3：重试 >= 2 次仍不通过

**动作**：
1. 停止 Workflow
2. 报告问题清单
3. 建议重新设计方案

## 七、与 pensieve 的协作

### 触发条件对比

| 维度 | karpathy-guidelines | Pensieve taste review | Pensieve self-improve |
|------|---------------------|----------------------|----------------------|
| 调用方式 | task-workflow 在代码变更阶段调用 | task-workflow 条件触发（变更 > 100行/新模块/数据结构重构） | task-workflow verification 后自动触发 |
| 检查粒度 | 函数/模块 | 系统/架构/接口品味 | 认知/模式/决策 |
| 执行频率 | 每次代码变更 | 0-2 次/Workflow | 0-1 次/Workflow |
| 失败后果 | 修正代码 | 返回重新实现 | 无（沉淀即完成） |

### 执行顺序

karpathy-guidelines 先执行，通过后才触发 pensieve（如果满足条件）。
