# task-workflow 执行规范

**版本**：v3.0
**最后更新**：2026-04-15

---

## 编排器职责边界

task-workflow 是编排器，职责范围：

| 编排器负责 | 编排器不负责 |
|-----------|-------------|
| 何时触发哪个技能 | 技能内部执行方法 |
| 步骤间数据传递（传递哪些字段） | 字段的格式化模板 |
| 质量门（通过/不通过） | 技能使用原则 |
| 失败回退路径 | 技能的操作规程 |

**重要**：技能如何执行，由技能自己的 SKILL.md 定义。编排器不侵入技能内部。

---

## 数据传递规范

### 传递格式

每个 Step 必须显式声明：
1. **读取输入**：从哪个 Step 读取哪些字段
2. **输出字段**：传递给下一个 Step 的字段名和类型

### 示例

```markdown
## Step 2: contradiction-analysis

**读取输入**（来自 Step 1）：
- 现状：string
- 关键约束：list<string>
- 关键发现：list<string>

**输出字段**（传递给 Step 3）：
- 主要矛盾：string
- 矛盾主要方面：string
- 解决策略：string
```

### 传递规则

1. **必须显式声明**：不得隐式传递
2. **必须按顺序执行**：不得跳过 Step
3. **必须完整传递**：不得遗漏关键字段
4. **字段类型明确**：string/list/dict/number

---

## 质量门标准

### 通用标准（适用于所有 Step）

编排器从外部判断技能是否完成：

1. **输出完整性**：
   - ✅ 所有必需字段都已输出
   - ✅ 字段类型符合定义

2. **置信度达标**：
   - ✅ 技能输出的置信度 >= 80%
   - ✅ 置信度有评估依据

3. **可传递性**：
   - ✅ 输出字段可以被下一个 Step 使用
   - ✅ 没有"待定"或"未知"的关键字段

### 特定 Step 标准

每个 Workflow 定义中，为每个 Step 指定：
- 必需输出字段
- 最低置信度要求
- 特定的完成条件

**注意**：这些标准是从编排器角度看的，不是技能内部的标准。

---

## 置信度机制

### 置信度来源

置信度由**技能自己**计算并输出，编排器只负责判断是否达标。

### 置信度阈值

- **>= 90**：高置信度，可直接推进
- **80-89**：中等置信度，标注风险后推进
- **< 80**：低置信度，触发失败回退

### 编排器行为

```markdown
if 置信度 >= 80:
    推进到下一个 Step
else:
    执行失败回退
```

---

## 失败回退机制

### 回退触发条件

1. **置信度不达标**：< 80%
2. **输出不完整**：缺少必需字段
3. **质量门未通过**：不满足特定条件
4. **循环次数过多**：>= 3 次

### 回退规则

| 触发条件 | 回退动作 |
|---------|---------|
| 置信度 < 80% | 返回当前 Step，要求补充证据 |
| 输出不完整 | 返回当前 Step，要求补充字段 |
| 质量门未通过 | 返回当前 Step 或上一个 Step |
| 循环 >= 3 次 | 停止 Workflow，报告给用户 |

### 回退格式

```markdown
[Step X 未通过质量门]

**未通过原因**：
- 置信度：XX% < 80%
- 缺少字段：[字段名]

**回退到**：Step X（补充证据）
```

## Workflow 执行流程

### 标准流程

```
1. 匹配 Workflow（根据任务特征）
    ↓
2. 读取 Workflow 定义
    ↓
3. 按顺序执行 Step：
    ├── 调用技能
    ├── 技能执行（内部逻辑）
    ├── 技能输出结果
    ├── 编排器检查质量门
    ├── 通过 → 传递数据到下一个 Step
    └── 不通过 → 失败回退
    ↓
4. 所有 Step 完成 → 输出 markdown 内容
    ↓
5. Workflow 完成
```

### 编排器的职责

- ✅ 调用技能（传递参数）
- ✅ 接收技能输出
- ✅ 检查质量门
- ✅ 传递数据
- ✅ 执行回退

### 编排器不做的事

- ❌ 不干预技能内部执行
- ❌ 不定义技能的输出格式
- ❌ 不定义技能的操作规程

## 最终输出标准

### 统一输出格式

所有 Workflow 的最终 Step 必须输出 **markdown 格式内容**，作为人类可读的最终交付物。

| Workflow | 输出内容 |
|----------|---------|
| W1: 通用执行 | 任务完成报告：需求/问题、主要矛盾、实践方法、实现摘要、品味评分、经验教训、验证证据、置信度 |
| W2: 新项目启动 | 项目启动方案：现状与约束、主要矛盾、根据地、路线图、完整战略、品味评分、置信度 |
| W3: 品味提升 | 品味审查报告：改造前后品味评分对比、致命问题修复、回归检查、证据、置信度 |

### 输出要求

1. **内容必须包含置信度** — 每个结论附带置信度数值
2. **证据必须可验证** — 引用具体代码位置、测试输出、数据指标
3. **格式统一** — 使用 markdown 标题层级、表格、代码块组织内容

---

## 铁律（编排层）

1. **调用顺序不可变** — 必须按 Workflow 定义的顺序调用技能
2. **质量门不可跳过** — 每个 Step 必须通过质量门才能推进
3. **数据传递不可省略** — 必须显式传递所有必需字段
4. **循环次数有上限** — 同一个 Step 循环 >= 3 次必须停止
5. **置信度不可造假** — 必须使用技能输出的真实置信度
6. **最终输出必须为 markdown** — 所有 Workflow 必须产出人类可读的 markdown 格式内容

## 与技能的协作

### 技能的职责

技能（如 investigation-first）负责：
- 定义自己的操作规程
- 定义自己的输出格式
- 计算自己的置信度
- 执行具体任务

### 编排器的职责

编排器（task-workflow）负责：
- 决定何时调用技能
- 传递输入参数
- 检查输出是否完整
- 判断置信度是否达标
- 决定是否推进或回退

### 协作示例

```markdown
编排器：调用 investigation-first，传递参数 {任务描述}
    ↓
技能：执行调查（按自己的操作规程）
    ↓
技能：输出结果 {现状, 约束, 发现, 置信度: 85%}
    ↓
编排器：检查质量门
    - 输出完整性：✅ 所有字段都有
    - 置信度：✅ 85% >= 80%
    ↓
编排器：传递数据到 Step 2
```

## karpathy-guidelines 质量门

### 调用时机

task-workflow 在代码编写阶段调用 karpathy-guidelines，作为编码规范检查点。

### 质量门标准

**通过条件**：
- ✅ 规范检查结果 = "通过"
- ✅ 置信度 >= 80%

**可修正条件**（触发重试）：
- ✅ 规范检查结果 = "不通过"
- ✅ 违反原则 <= 2 项
- ✅ 置信度 >= 80%
- ✅ 具体问题有明确修改建议

**严重失败条件**（停止 Workflow）：
- ❌ 违反原则 >= 3 项
- ❌ 置信度 < 80%
- ❌ 重试 >= 2 次仍不通过
- ❌ 发现架构级问题（需重新设计）

### 执行逻辑

```
if 规范检查结果 == "通过":
    推进到下一个 Step
elif 规范检查结果 == "不通过" and 违反原则 <= 2 and 置信度 >= 80:
    应用修改建议 → 重新检查（最多 2 次）
else:
    停止 Workflow → 报告给用户
```
